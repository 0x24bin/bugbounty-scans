User Agent: Mozilla/5.0 (X11; Ubuntu; Linux x86_64; rv:27.0) Gecko/20100101 Firefox/27.0 (Beta/Release)
Build ID: 20140218135856

Steps to reproduce:

I've made a webpage with two language selectors (one for mobile and one for desktop version, only one of them is visible at a time). The mobile language selector works well, but the desktop one freezes. They're identical, except the font size. 

I've found the root of the problem. Pulldown (HTML "select" elm) freezes when the following condition meets:
- the page is UTF8 encoded (meta content type set to UTF8)
- font-size is specified (with CSS or HTML attrib, e.g. style="font-size: 12px")
- have at least one option item with contains Arabic letters (as text value, not as key)
- it's the first pulldown with these conditions.

So: first pulldown contains arabic letters and have font size set freezes. Sounds rare, but if you're using pulldown for language selector, and you have arabic language on your site, it's essential.

Not as important, but still a bug, that if there is Arabic font in the pulldown, there is
- cca. 1 line height padding inside the closed state pulldown, at bottom of it,
- cca. 5-6 lines height padding inside the opened state pulldown, at bottom.

See attached HTML code, there are 4 pulldowns in it:
- font size set, contains Arabic item: padding, freezes
- font size not set, contains Arabic item: padding, no freeze
- font size set, contains no Arabic item: no padding, no freeze
- font size set, contains Arabic item (just as the first one, which freezes): padding, no freeze(!)


Actual results:

On collapsed state, the buggy pulldown displays the first option, as usual. Clicking on it, nothing happens, it does not expands, and Firefox eats up 85% of processor (reported by "top" command).


Expected results:

The pulldown should filfull it's destiny: the options should pull down.
